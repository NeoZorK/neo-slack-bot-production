## Overview

Version 0.0.7 focuses on **code quality improvements**, **documentation compliance**, and **enhanced reliability** based on the architecture audit findings and production testing feedback.

---

## Goals

1. **Fix All Audit Issues**: Address all findings from architecture audit
2. **Documentation Compliance**: Ensure all documentation files comply with 300-line limit
3. **Enhanced Reliability**: Improve error handling and recovery mechanisms
4. **Performance Optimization**: Optimize critical paths based on profiling

## Priority 4: Cross-Platform and Deployment

### 4.1 Linux and Windows Support

**Goal**: Make core bot functionality runnable on Linux and Windows (headless mode, no GUI dependencies).

**What is already done**:
- Linux implementations for notifications and system metrics are in place and documented in `linux-support.md`
- Windows stub implementations for notifications and system metrics are in place and documented in `windows-support.md`
- Docker-based Linux image exists and can run the full test suite in headless mode

**Scope (for v0.0.7 end-game)**:
- Treat Linux as a reference CI platform alongside macOS
- Ensure `test_neo_slack_bot` passes on at least one reference Linux distribution
- Validate notification flow using libnotify in a real desktop session
- Ensure core event loop, filtering, and HTTP/WS clients build and run on Windows
- Keep macOS-specific features (native notifications, Keychain) optional and gracefully degraded on other platforms
- Document platform-specific limitations and setup for each supported OS

**Technical checklist**:
- [ ] Project builds successfully on a reference Linux distribution using CMake + Ninja (Release + Coverage)
- [ ] `test_neo_slack_bot` binary runs and all non-macOS-only tests pass on Linux
- [ ] Linux notifications backend is exercised by tests or manual scenario and documented with concrete scenarios in `linux-support.md`
- [ ] Project builds successfully on Windows (MSVC or MinGW) with current stub implementations
- [ ] Core bot binary starts on Windows and can process a minimal event flow (no GUI required)
- [ ] Windows stub notifications (console logging) are verified with a simple scenario and documented in `windows-support.md`
- [ ] Platform support docs (`linux-support.md`, `windows-support.md`) are updated for v0.0.7 with known limitations and setup steps

### 4.2 Docker Support

**Goal**: Provide first-class Docker support for Linux-based testing and CI (not production).

**Scope (for v0.0.7 end-game)**:
- Finalize Docker image build for Linux-based testing with CMake + Ninja pipeline
- Keep Docker images focused on core functionality (WebSocket, HTTP API, JSON parsing, metrics)
- Explicitly document limitations due to macOS-only features (notifications, Keychain) in Docker docs
- Integrate Docker usage into testing and CI workflows as an optional but recommended path
- Update Docker docs to reflect v0.0.7 status and cross-platform expectations

**Technical checklist**:
- [ ] Docker image builds successfully from project root on a reference Linux host
- [ ] Standard test suite runs inside Docker and completes without errors (excluding macOS-only tests)
- [ ] Environment-variable based tests (tokens, config path) are documented and verified for Docker runs
- [ ] Performance or stress tests have documented Docker invocations (where applicable)
- [ ] Docker usage is integrated into at least one CI example (GitHub Actions or GitLab CI) using the provided Dockerfile
- [ ] Docker docs (`docker.md`, `docker-setup.md`, `docker-usage.md`) describe v0.0.7 expectations and limitations, including Linux-only scope and macOS feature gaps

**Estimated Effort**: 6-8 hours
